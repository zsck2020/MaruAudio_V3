# MaruAudio V3 开发规则

## 代码读取规则

### 使用 augment-context-engine 工具

在读取代码文件时，**必须**调用 `augment-context-engine` 工具。

**原因**：
- `augment-context-engine` 使用服务器语言协议（LSP）而不是简单的文本搜索
- 能够提供精确的代码语义理解
- 可以分析代码结构、依赖关系、类型信息等

**执行方式**：
```bash
auggie --mcp --mcp-auto-workspace
```

**何时触发**：
- 使用 Read 工具读取任何代码文件时
- 分析代码结构或查找符号定义时
- 需要理解代码语义而非文本内容时

**已配置自动触发**：
已在 `~/.claude/settings.json` 中配置 PreToolUse hook，当 Read 工具被调用时会自动执行 augment-context-engine。

## 任务管理规则

使用 `egughjls_cov` 的 MCP 工具来进行团队任务管理。

### 任务流程
1. 分析任务需求
2. 输出任务进展
3. 调用 `egughjls_cov`
4. 接收下一步指令
5. 推进任务

### INPUT 参数
- `destinationPanelId` → 目标面板ID，多窗口指定向哪个面板发送请求
- `prompt` → 询问用户下一步想要做什么（默认："下一步想做什么？"）
- `summary` → 当前对话的上下文摘要，了解 AI 已完成的工作
- `options` → 可选，供用户选择的选项列表

### OUTPUT 参数
- `reply` → 用户的输入，必须仔细遵守该字段的所有内容
- `directive` → `end_loop`（下次不需调用mcp了）/ `continue`（下次结尾调用`mcp`）
- `sourcePanelId` → 响应来自的面板ID，下次调用时传入参数 `destinationPanelId`

### 工作准则
- **实事求是**：不确定就坦诚告知
- **根本解决**：分析问题本质
- **整体考量**：编辑前检查关联
- **善用工具**：充分利用 IDE 能力

## 优先规则 (MANDATORY)

### 规则 1：结束前询问

**每次回复将要结束时，倾向于提供 2~4 个后续选项供用户选择，而不是直接结束对话。**

- 在每次回复的最后，提供与当前上下文相关的后续选项
- 选项应与当前任务相关，不要每次都用固定选项
- 等待用户从选项中选择后再继续

**示例场景**：
- 代码编写完成后：「继续下一个功能」「测试运行」「代码审查」「结束」
- 问题诊断中：「继续排查」「尝试修复」「换个方案」「结束」
- 方案讨论后：「开始实现」「调整方案」「需要更多信息」「结束」

---

### 规则 2：需求确认

**每次收到新需求时，先向用户详细复述对需求的理解，等用户确认后再执行。**

**要求**：
- **先复述理解**：用自己的话详细描述对需求的理解，包括：要做什么、为什么做、预期结果
- **等待确认**：让用户确认理解是否正确
- **确认后执行**：只有用户确认理解正确后，才开始实际执行
- **有歧义时追问**：如果需求有多种理解方式，列出不同理解让用户选择

**例外情况**（可直接执行，无需确认）：
- 用户明确说"直接做"或"不用确认"
- 非常简单明确的操作（如"读取某文件"、"运行某命令"）

---

### 规则 3：先验证再行动

**遇到问题或需求时，先分析验证，精准定位问题，再针对性解决。避免先入为主假设原因。**

**执行流程**：
1. **收集信息** → 读取相关代码/日志/数据，了解当前真实状态
2. **分析验证** → 用事实数据确认问题根因，而非假设
3. **制定方案** → 基于确认的根因，制定最小化修复方案
4. **执行修复** → 针对性实施
5. **验证结果** → 确认修复有效

**反面示例**（不推荐）：
- "问题可能是 X，让我先改代码试试" → 假设驱动
- 连续创建多个脚本尝试不同方案 → 盲目试错

**正面示例**（推荐）：
- "让我先查看日志/数据库确认当前状态" → 事实驱动
- "根据日志确认根因是 Y，针对性修复" → 精准定位
